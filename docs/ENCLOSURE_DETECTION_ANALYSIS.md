# 囲い込み（Enclosure）判定の分析

## 座標系

- **左上が (0, 0)** の0始まり
- 本ドキュメントでは **(行, 列)** = **(row, col)** の順で表記する（例: (2,1) = 行2・列1）
- `board-worker.js` の path は `{ x: 列, y: 行 }` を採用（x＝横・列、y＝縦・行）
- x軸・y軸どちらも 0 から開始し、左上は (0, 0)

### 用語「縦」「横」は **画面上の向き** と **`path` のどの成分が動くか** をセットで指す

本ドキュメントおよびコードコメントの **「縦」「垂直」** は、**画面上で上下に伸びる辺**の意味であり、かつ **`path` では列 `x` が一定で行 `y` だけが変わる移動**（`dx = 0`, `dy ≠ 0`）を指す。**数学・グラフィックスでいう「Y 方向の増減」＝ この `y`（行インデックス）の増減** と一致する（行番号が増える向き＝通常は画面では下向き）。

- **縦セグメント / 垂直** … **同じ列（`x` 固定）** を **行（`y`）が変わる**ように進む辺。`classifyColumnPassSide`・**垂直囲い込み（列 `nc`）**・`refDy` はこの「列に沿った上下」の話。

**「横」「水平」** は **画面上で左右に伸びる辺**であり、**`path` では行 `y` が一定で列 `x` だけが変わる移動**（`dy = 0`, `dx ≠ 0`）を指す。

- **横セグメント / 水平** … **同じ行（`y` 固定）** を **列（`x`）が変わる**ように進む辺。`classifyRowPassSide`・**水平囲い込み（行 `nr`）**・`refDx` はこの「行に沿った左右」の話。

**表記のずれに注意:** 日本語で **(行, 列)** の順と、`path` の **{ x: 列, y: 行 }** の **(列, 行)** 順が**逆**である。文に「縦」とだけあるときは **「`y`（行）が動く」＝ 列筋の上下** と読み替えるとコードと一致する。

## アルゴリズムの概要

`board-worker.js` の `countPairLinkEnclosures` / `analyzePairLinkEnclosuresDebug` は、以下の条件を**すべて**満たしたときにのみ「囲い込み」としてカウントする。

### 1. ターゲットは端点のみ

```javascript
if (adj[nr][nc].length !== 1) continue;  // degree === 1 のセルのみ
```

- **囲まれる側（被囲い込み）**は必ずパスの**端点**（数字マス）
- パスの中間セルはターゲットにならない

### 2. 幾何的ブラケット（挟み）

- **垂直囲い込み**: 囲むパスが「列 nc」上に `y1 < ターゲット行 < y2` となる2点を持つ  
  - **優先**: ターゲット行を挟む **直近ブラケット** — `y < tRow` 側では **y 最大**、`y > tRow` 側では **y 最小** のセル（同値はパス上でより前のインデックス）
  - **フォールバック**: 上記が取れないとき従来どおり列上の **y 全体の最小・最大**
- **水平囲い込み**: 囲むパスが「行 nr」上に `x1 < ターゲット列 < x2` となる2点を持つ  
  - **優先**: `x < tCol` 側で **x 最大**、`x > tCol` 側で **x 最小**（同値はより前のインデックス）
  - **フォールバック**: 行上の **x 全体の最小・最大**
- **pseudo 窓**: 角から続くセグメントは **最大 4 本**まで見る（K=4）

### 3. 追い越し方向が逆であること（最重要）

- `classifyColumnPassSide` / `classifyRowPassSide` で、ブラケット2点それぞれについて、パスがその列/行の「どちら側から来たか」を判定
- **垂直**: 2点のうち一方が `"L"`（左側から）、もう一方が `"R"`（右側から）でなければならない
- **水平**: 2点のうち一方が `"B"`（下側から）、もう一方が `"A"`（上側から）でなければならない
- **同じ方向**（例: 両方 `"B"`）の場合は `same_x_pass_direction` / `same_y_pass_direction` で除外

### 4. pseudo U-turn（エセ回り込み）除外（ブラケット基準・現行）

- **垂直囲い込み**: 列 `nc` 上のブラケット点 `iMin` / `iMax`（**ターゲット挟み優先**、§2 参照）について、当該点でパスが列上に持つ**垂直セグメント**の向き（`refDy`）を取り、そのセグメントの**角**（`cornerIdx`）から**最大4セグメント**先まで進み、**同じく列 `nc` 上の垂直セグメント**で `refDy` と**符号が逆**のものがあり、かつターゲット行の近傍（±2 行）に関係すれば pseudo とみなす
- **水平囲い込み**も同様に、行 `nr` 上のブラケットで水平 `refDx` と**逆向きの水平**を**最大4セグメント**以内で探す

### 5. 同一ペアの複数端点（`countPairLinkEnclosures` / `analyzePairLinkEnclosuresDebug`）

- 被囲い側パス `qPid` の **各端点セル** `(nr, nc)` について、いずれも `checkVerticalWrapEnclosure` / `checkHorizontalWrapEnclosure`（＋ pseudo）を実行する。
- **囲む×囲まれるの組** `pPid + ">" + qPid` について、**囲い込み件数は従来どおり最大 1**（先に確定した端点で `pairCounted` したあと、もう一方の端点ではカウント・Found 可視化を重ねない）。pseudo 可視化・ログも同じキーで 1 回に抑える。

## ケース1: パス4がパス6の端点 (1,1) を囲い込む場合の検証

### 見た目

- パス4は左上付近を「C字形」で囲んでいるように見える
- パス6の端点 (1,1) が、その囲いの内側にあるように見える

### 幾何的ブラケット条件の検証

ターゲット: パス6の端点 (1,1)（行1・列1）

**水平囲い込み（行 nr=1 上で x1 < 1 < x2）**

- パス4が行1上に存在する点を列挙する
- パス4の全経路: (4,0)→(3,0)→(2,0)→(1,0)→(0,0)→(0,1)→…→(0,6)→(1,6)→(1,5)
- 行1上の点: **(1,0), (1,6), (1,5)** → xMin=0, xMax=6
- ターゲット列1: `0 < 1 < 6` → **幾何的ブラケット条件は満たす**

**追い越し方向の判定**

- 左ブラケット (1,0): パスは (2,0)→(1,0)→(0,0)。y≠1 の点を探すと (2,0) が y=2>1 → `"A"`（上側から）
- 右ブラケット (1,6): パスは (0,6)→(1,6)→(1,5)。y≠1 の点を探すと (0,6) が y=0<1 → `"B"`（下側から）
- `"A"` と `"B"` は逆 → **追い越し方向の条件は満たす**

**pseudo U-turn チェック**

- 行1への進入: (2,0)→(1,0) で上から、または (0,6)→(1,6) で下から
- 進入後 3 セグメント以内に、逆向きの dy で行1へ戻る動きはない
- pseudo U-turn には該当しない

### 結論（(1,1) の場合）

- 幾何的ブラケット・追い越し方向・pseudo のいずれも通過する
- **パス4はパス6の (1,1) を囲い込みとしてカウントされる**

## ケース2: パス2がパス3の端点 (5,4) を囲い込んでいるように見えるがカウントされない理由

### 見た目

- パス2は (4,2)→(4,5) で右へ伸び、(6,5) で下がり、(6,4) で**左へ戻り**、(7,4)→(7,5) で下・右へ
- パス2の折り返しが (5,4) の「外側」を形作り、囲んでいるように見える

### 幾何的ブラケット条件の検証

ターゲット: パス3の端点 (5,4)（行5・列4）

**垂直囲い込み（列 nc=4 上で y1 < 5 < y2）**

- パス2が列4上に存在する点を列挙する
- パス2の経路: (4,2)→(4,3)→(4,4)→(4,5)→(5,5)→(6,5)→(6,4)→(7,4)→(7,5)
- 列4上の点: **(6,4), (7,4)** → yMin=6, yMax=7
- ターゲット行5: `yMin < 5 < yMax` を満たすには `6 < 5 < 7` が必要だが、**6 < 5 は偽**
- ターゲット (5,4) は行5にあり、パス2の列4上の点は行6・7のみ → **`target_not_between_y`**（ターゲットが y の範囲外）

**水平囲い込み（行 nr=5 上で x1 < 4 < x2）**

- パス2が行5上に存在する点を列挙する
- 行5上の点: **(5,5) のみ**（(4,5)→(5,5)→(6,5) の通過点）
- 同一行上に2点以上必要だが1点のみ → **`no_row_span`**

### 結論

- パス2が列4で通るのは (6,4) と (7,4) の2点のみで、いずれも行5より**下**（行6・7）
- ターゲット (5,4) は行5のため、アルゴリズムが要求する「ブラケット2点の y の**間**にターゲットがある」を満たさない
- 見た目ではパス2が (5,4) を囲んでいるように見えても、**幾何的ブラケット条件（ターゲットが2点の間にある）** を満たさない時点で除外される

---

## 補足: パス2の「引き返し」が pseudo U-turn で除外されない理由

パス2は (6,5)→(6,4) で左へ戻るが、pseudo U-turn には該当しない。

- pseudo U-turn は「**行／列に垂直に進入した直後、その行／列へ戻る動き（逆向き dy/dx）**」を検出する
- 行6への進入 (5,5)→(6,5) 後の 3 セグメント: (6,5)→(6,4) は水平（dy=0）、(6,4)→(7,4) は下方向（dy=1）
- 行6へ戻る上方向（dy=-1）の動きは存在しない
- そのため、「進入→即Uターン」とは判定されず、pseudo U-turn には引っかからない

## コード上の原因分析と修正提案（パス4→6がカウントされない問題）

### 原因: パス traverse 開始点による `classifyRowPassSide` の方向依存

**1. `getOrderedPathForPid` の開始端点選択**

`getOrderedPathForPid` は `r=0,c=0` から row-major で走査し、**最初に見つかった**端点（`adj[r][c].length === 1`）から traverse を開始する。

- パス4の端点: (4,0) と (1,5)
- (1,5) が (4,0) より先に見つかる → **常に (1,5) から開始**
- その結果、パス順は `(1,5) → (1,6) → (0,6) → … → (0,0) → (1,0) → (2,0) → … → (4,0)`

**2. `classifyRowPassSide` の「後方優先」**

`classifyRowPassSide(path, idx, ty)` は、`path[idx]` について **後方（idx-k）を先に** 探し、`y !== ty` となる最初の点で "A" または "B" を返す。

- 左ブラケット (1,0): 後方に (0,0)（y=0<1）→ `"B"`
- 右ブラケット (1,6): 後方に (1,5)（y=1=ty）はスキップ、次に (0,6)（y=0<1）→ `"B"`
- 両方 `"B"` → `same_y_pass_direction` で除外

**3. 逆向きパスにすると正しく判定される**

パスが `(4,0) → … → (1,0) → (0,0) → … → (1,6) → (1,5)` のとき:
- (1,0): 後方に (2,0)（y=2>1）→ `"A"`
- (1,6): 後方に (0,6)（y=0<1）→ `"B"`
- `"A"` と `"B"` → 追い越し方向が逆となり、囲い込みとしてカウントされる

つまり、**幾何的には正しく囲めているのに、開始端点の選び方でパス向きが決まり、classify の結果が変わる**のが原因。

---

### 修正案

**案A: パス反転のフォールバック**

`sLeft === sRight` のとき、パスを反転した配列に対してもう一度 `classifyRowPassSide` を実行する。反転後の path では、同一座標の点のインデックスが `len - 1 - idx` になる。反転後の `sLeft'`, `sRight'` が opposite なら囲い込みと判定する。

- 利点: 既存ロジックの拡張で、他への影響が小さい
- 欠点: 水平・垂直の両方で同じフォールバックが必要

**案B: `getOrderedPathForPid` の開始点を正規化**

開始端点を「行・列の和が大きい方」や「(r,c) の辞書順で後ろ」など、一貫したルールで選ぶ。パス4では (4,0) を選ぶよう調整する。

- 利点: classify 側は変えなくてよい
- 欠点: 他のパスで逆に問題が出る可能性があり、正規化ルールの選び方が難しい

**案C: classify を「双方向考慮」に変更**

`classifyRowPassSide` で、後方・前方の**両方**から `y !== ty` の点を集め、「上側を通るか・下側を通るか」を両方向の情報で決める。例えば、後方と前方で異なる側（上と下）を通っていれば、その点は「行を横切っている」とみなし、ブラケット2点の組み合わせで opposite 判定に使う。

- 利点: パス向きに依存しない
- 欠点: classify の仕様変更が大きい

**案D: 「両向きパス」で両方を試す**

`checkHorizontalWrapEnclosure` 内で、元パスで `sLeft === sRight` の場合に、path を反転した配列を新規作成し、その path に対して同じ iMin, iMax に相当するインデックス（`len-1-iMin`, `len-1-iMax`）で再度 classify する。反転後で opposite なら `ok: true` とする。

- 利点: check 側のみの変更で済む
- 欠点: パス配列のコピーと反転のコスト

---

### 推奨

**案A / 案D**（いずれも「sLeft === sRight のときに反転パスで再判定」）が、変更範囲が限定的で実装しやすい。垂直・水平の両方の `check*WrapEnclosure` に同様のフォールバックを入れる。

---

## 設計意図の整理

| 条件 | 目的 |
|------|------|
| 追い越し方向が逆 | 「左右」「上下」両方から本当に回り込んでいる経路だけを囲い込みとする |
| pseudo U-turn 除外 | わずかな凹凸で「見かけ上」だけ囲んでいる経路を除外する |

その結果として、

- 「C字形」で一方からだけ回り込んでいる経路 → カウントされない
- 「進入→即Uターン」していない、素直な横切りの囲い込み → カウントされる

という挙動になっている。

## ケース3: パス2（青）がパス3（緑）の端点 (6,6) を「垂直囲い込み」としてカウントする理由（添付・修正後盤面）

ユーザー報告の盤面では、**パス4→パス6** の horizontal pseudo は紫の点線＋▼▲で正しく棄却される一方、**パス2→パス3** の **(行6, 列6)** 周りは **黄色の縦破線**（有効な垂直囲い込み）のままになる。ここでは **アルゴリズムが意図どおり通過する条件** を、座標と経路に沿って分解する。

### 座標・表記の対応

- 盤面の説明で用いる **(x, y) = (列, 行)**（右・下が正）は、コードの `path[i] = { x, y }` と一致する
- 本ドキュメントの **(行, 列)** で書く場合は **(y, x)** に相当する（例: ターゲットの緑「3」が **(行6, 列6)** → `path` では `{ x: 6, y: 6 }`）

### 前提（ユーザー盤面と同一とされる経路）

**囲む側・パス2（青）**の端点と通過（列・行表記）:

- 端点の一例: **(列3, 行5)** と **(列8, 行9)**
- 経路の骨格（セル中心のグリッド移動として）:  
  **(3,5)→(7,5)**（行5を右へ）→ **(7,7)**（列7で下へ）→ **(6,7)**（左へ）→ 以降下側へ抜けてもう一方の端点へ  

※ 実際の `path` 配列には中間マス（例: (7,6) など）が挟まるが、**列6上に現れる行番号**だけが判定に効く。

**囲まれる側・パス3（緑）**:

- ターゲットは **degree 1 の端点** として **(行6, 列6)**、すなわち `{ x: 6, y: 6 }`（破線がこのマスを貫く）

### 1. 垂直囲い込みの幾何（`checkVerticalWrapEnclosure`）

パラメータ: **nc = 6**（列）、**tRow = 6**（ターゲットの行）。

パス2が **列6** 上に載るマスを列挙すると、典型的には少なくとも:

- **(列6, 行5)** … 行5を (3,5)→(7,5) と進む区間
- **(列6, 行7)** … (7,7)→(6,7) の左進入の先

の2点を含む（経路によっては **(列6, 行8)** などさらに下の点が続く場合もあるが、以下の不等式の結論は同じ構造になる）。

アルゴリズムは列6上の **最小行 yMin** と **最大行 yMax** を取る:

- 上記2点のみなら **yMin = 5**, **yMax = 7**
- **(6,8)** まで列6にいるなら **yMax = 8**

いずれの場合もターゲット行 **6** は **yMin < 6 < yMax** を満たす（例: `5 < 6 < 7` または `5 < 6 < 8`）。  
→ **`target_not_between_y` にはならず、幾何的ブラケットは満たす。**

（デバッグの破線は **yMin〜yMax** の縦線なので、経路に (6,8) があると下端が行8まで伸びる。画像が「(6,5)〜(6,7)」に見える場合は、盤面によっては yMax=7 の経路か、見た目のスケールの差として読める。）

### 2. 追い越し方向（`classifyColumnPassSide`）が通る理由

ブラケットは **列6上で y が最小のインデックス iMin** と **最大の iMax**（コードは走査順で最初に更新されたインデックス）。

- **下側ブラケット**（例: **(6,5)**）  
  - パスが (5,5)→(6,5)→(7,5) のように通ると、`(6,5)` から **後方**に進んで最初に **x ≠ 6** となる点は **(5,5)**（x=5）→ **`"L"`**
- **上側ブラケット**（例: **(6,7)**）  
  - `(6,7)` の直前が **(7,7)**（x=7）なら、後方で最初の **x ≠ 6** は **(7,7)** → **`"R"`**

**`"L"` と `"R"` は逆**なので、追い越し方向の条件を満たす。  
（もし偶然どちらも同じ側になった場合は、実装済みの **パス反転フォールバック**で再判定される。）

→ ユーザーが「見た目は青線が一度右に伸びてから折り返して戻っている」と感じる形状でも、**ブラケット2点では「列6を左から触れている」と「右から触れている」が別**になっており、**アルゴリズム上は正当な「両側からの挟み」**として扱われる。

### 3. ブラケット基準 pseudo（`isVerticalBracketPseudoEnclosure`）に引っかからない理由

pseudo は **「ブラケット上の垂直な腕の向き `refDy`」** と **「角から続く3セグメント以内の、列6上の逆向き垂直セグメント」** の組み合わせで検出する。

#### 3a. 下側ブラケット (6,5)

- `(6,5)` では隣が **(5,5)** と **(7,5)** のように **どちらも行5** であり、**列6に沿った垂直セグメント（dy≠0 かつ両端 x=6）が存在しない**
- `pickVerticalBracketRef` が **null** → このブラケットからは pseudo 判定を開始できない

#### 3b. 上側ブラケット (6,7)（および列6がさらに下へ続く場合）

- 列6上の垂直セグメントは **(6,7)→(6,8)** のような **下向き（dy>0）** が典型
- **yMax 側のブラケット**では `refDy < 0`（上向きの「腕」）を優先するが、なければ **実際にあるセグメント（下向き）** を `ref` に使う
- **角 `cornerIdx`** はその垂直セグメントの終端側（例: **(6,8)**）に置かれ、そこから先の **最大3セグメント** を見る
- 続く辺は **(6,8)→(7,8)** のような **水平** や、さらに先の **下向きの垂直** が続くことが多く、**「列6上で refDy と符号反対の垂直セグメント」** が **3セグ以内・かつターゲット行6の近傍条件**を同時に満たさない

したがって **「角で折り返した直後のエセ縦往復」** パターンとしては検出されず、**pseudo では棄却されない**。

### 4. まとめ（なぜ「逆戻り」に見えてもカウントされるか）

| チェック | 結果 |
|----------|------|

→ **この盤面では「パス2がパス3の (6,6) を垂直囲い込みとしてカウントされる」ことは、現行ルールに照らして一貫している。**  

ユーザーが期待する「大きなUターン全体を逆戻りとみなす」判定は、**今の pseudo は「ブラケット近傍の短い縦往復」に特化**しており、**(7,7)→(6,7)** のような **水平での入射＋別方向への離脱** までは **同じチェックでは除外しない**、という設計上の隙間として説明できる。

## ケース4: ユーザ期待（パス2×「3の端点 (5,4)」）と現行実装の齟齬

ユーザーは次のように、**回り込みが棄却されてほしい**理由を説明している。

- **パス2（列・行 = `path` の x,y）:**  
  `(4,2)→(4,3)→(4,4)→(4,5)→(5,5)→(6,5)→(6,4)→(7,4)→(7,5)`
- **囲いのイメージ上の2点:** **(4,4)** と **(6,4)**（いずれも行4）
- **追い越しベクトル（ユーザ表現）:** `(4,4)→(4,5)` が右(R)、`(6,5)→(6,4)` が左(L)
- **Uターン（ユーザ表現）:** `(6,4)` から数えて3セグメント以内の `(7,4)→(7,5)` の「右(R)」が、`(6,5)→(6,4)` の「左(L)」と対になり **回り込み棄却** すべき

以下、**現行 `board-worker.js` と照合**して齟齬を列挙する。

### 注: ターゲット `(5,4)` の解釈

- 本リポの **ケース2** では **(5,4) = (行5, 列4)** → `{ x:4, y:5 }` と読み、**幾何的ブラケットで落ちる**話になっている。
- ユーザの **「(4,4) と (6,4) で挟む」** という説明は、**ターゲットが列5・行4**、すなわち **`{ x:5, y:4 }`** のときの **水平囲い込み（行 nr=4, tCol=5）** と整合する。
- 以下は **ターゲット = `{ x:5, y:4 }`** を仮定して実装を追う（`(行5,列4)` の場合はそもそも水平・垂直とも別理由で成立しない）。

### 齟齬1: 「回り込みの2点」— 旧実装は **行上の最小・最大 x** であり **(6,4) ではない**

**（更新後）** `findHorizontalStraddleBrackets` により **ターゲット列を挟む直近2点**（左: `x<tCol` で最大 x、右: `x>tCol` で最小 x）を優先し、ユーザ想定の **(4,4) と (6,4)** に一致しやすくなった。取れない場合のみ従来の **行全体の min/max x** にフォールバックする。

以下は **当時の min/max ブラケット** に基づく説明のメモである。

行4上のパス2の点は **(4,4), (6,4), (7,4)** の **3点**。

- **xMin = 4** → **(4,4)**
- **xMax = 7** → **(7,4)**（ユーザが想定した **(6,4) ではない**）

したがって **「囲い込みと判断される2点」** はコード上 **(4,4) と (7,4)**。  
ユーザが **(4,4) と (6,4)** を対にして語る部分は、**アルゴリズムの定義と一致しない**。

### 齟齬2: 「2ベクトルが R と L」— 実装は **行に直交する側** の **B/A**（`classifyRowPassSide`）

囲い込みの通過判定は **`classifyRowPassSide(path, idx, ty=4)`**：

- 行 `ty` 上の点 `path[idx]` から、パス上で **最初に `y !== ty` となる点**を後方→前方で探し、**`y < ty` → `"B"`**、**`y > ty` → `"A"`** を返す（コメント上「下側／上側」）。

例（パス先頭が (4,2) のとき）:

- **(4,4):** 後方 (4,3) は `y=3<4` → **`"B"`**
- **(7,4):** 後方 (6,4) は同じ行のためスキップ、さらに (6,5) は `y=5>4` → **`"A"`**

**`"B"` と `"A"` は逆**なので、**追い越し方向の条件は通る**（ユーザの R/L ラベルとは **別物**）。

ユーザが挙げた **`(4,4)→(4,5)`** は **列固定の縦移動（dy>0）** であり、**行4上の水平ベクトルではない**。  
**`(6,5)→(6,4)`** も **行4上の辺ではない**。  
→ 実装の **B/A** は「行の**上か下か**からパスが来るか」で、**ユーザの「水平の R/L ペア」とは対応していない**。

### 齟齬3: Uターン検出 — 実装は **「行 nr 上の水平セグメント」の refDx** と **同じ行上の逆向き水平**のみ

`isHorizontalBracketPseudoEnclosure` → `pickHorizontalBracketRef` は:

- ブラケット点 `iBracket` について、**隣接2点とも `y === nr`** である辺だけを **水平セグメント**として集める（```653:661:public/workers/board-worker.js``` 付近）。

**右ブラケット (7,4)（xMax 側）:**

- (6,4)–(7,4) は **y=4 で両方行4** → **dx = +1** が ref 候補
- (7,4)–(7,5) は **y が変わる** → 行4上の水平としては **含めない**

**`atMax` では `dx < 0` を優先**するが、行4上で左向きの辺が無いため **実際に選ばれるのは dx>0**（(6,4)→(7,4)）。**`cornerIdx` は (7,4) のインデックス**（セグメントの `to`）。

**左ブラケット (4,4)（xMin 側）:**

- 前後は (4,3) と (4,5) で **どちらも y≠4** → **行4上の水平辺が1本も無い**
- **`pickHorizontalBracketRef` は null** → このブラケットからは pseudo 探索を開始できない

**その後 `tryHorizontalBracketCornerPseudo`:**

- `cornerIdx` から先 **最大3セグメント**について、**`dy === 0` かつ `dx` が refDx と符号反対**の辺だけを pseudo とする。

`(7,4)` からの次の辺は **(7,4)→(7,5)** で **垂直** → **`dy !== 0` のため無視**。その先も **行4上で refDx(+1) と逆向きの水平**は出てこない。

**結果: pseudo は false。**

### 齟齬4: ユーザが言う Uターン — **縦2本 `(6,5)→(6,4)` と `(7,4)→(7,5)`** は **水平 pseudo の対象外**

ユーザは **左向き・右向き**のように、**縦移動のペア**で Uターンを定義している（`(6,5)→(6,4)` と `(7,4)→(7,5)`）。

現行の **水平囲い込み用 pseudo** は:

1. **参照ベクトル**も **検出ベクトル**も **必ず「その行 nr 上の水平辺」（dy=0）** に限定される  
2. **`(6,5)→(6,4)`** は **列6上の縦**なので、**水平ブラケットの ref には使われない**  
3. **`(7,4)→(7,5)`** は **縦**のため、**「refDx に対する逆向き水平」**としては **そもそも評価されない**

つまり **ユーザが指摘している棄却理由は「行に直交する短い往復」** に近いが、**実装は「行に沿った水平のすぐ引き返し」だけ**を見ている。**軸が一致していない**。

### 齟齬5: 「(6,4) から3セグメント」— 実装の起点は **`cornerIdx` = (7,4)** 側

ユーザは **(6,4) を角**として 3 セグメントを数える。

実装は **xMax ブラケット**で選んだ **行内水平の終端**を `cornerIdx` にしており、この経路では **(7,4)**。  
**カウント開始位置がユーザ想定とずれている**。

### まとめ表（期待 vs 実装）

| 観点 | ユーザ期待に近い説明 | 現行実装 |
|------|----------------------|----------|
| ブラケット2点 | (4,4) と (6,4) | 行4上の **min/max x** → **(4,4) と (7,4)** |
| 通過の「逆」 | (4,4)–(4,5) と (6,5)–(6,4) を R/L のように見る | **行の上下**から来たか **B/A**（別定義） |
| Uターン | 縦ベクトル同士の対 `(6,5)–(6,4)` vs `(7,4)–(7,5)` | **同一行上の水平 refDx** と **続く3辺以内の逆向き水平**のみ |
| 3セグメントの起点 | (6,4) | **pick の `cornerIdx`**（本経路では **(7,4)**） |

### 結論

ユーザが述べる棄却ロジックが **実装と噛み合わない主因**は次の3点である。

1. **幾何ブラケット**が **(6,4) ではなく (7,4)** が右端になる（同一行に複数点があるため）。  
2. **囲い込みの「逆方向」**は **R/L ではなく B/A** で、**ユーザが引用した縦辺2本とは無関係**に判定される。  
3. **pseudo** は **「行に沿った水平の即座の逆向き」**のみを見ており、**縦の U 字 `(6,5)–(6,4)` と `(7,4)–(7,5)` はスコープ外**。

**棄却をコードで再現するには**、例として **(i)** 行nr上のブラケットを **「ターゲット列を挟む隣接2点」に限定する**、**(ii)** 角 (6,4) 近傍で **縦セグメント同士の符号反転**を pseudo に含める、**(iii)** xMax が **ターゲットより右に「突出」している**場合は別扱いする、など **仕様追加**が必要になる。
